### 2.3. Script Analysis (P2PKH)

#### 2.3.1. Structure of `scriptPubKey` (Locking Script)

For a P2PKH transaction, the `scriptPubKey` (also known as the locking script or challenge script) has the following structure:

`OP_DUP OP_HASH160 <PubKeyHash> OP_EQUALVERIFY OP_CHECKSIG`

- `OP_DUP`: Duplicates the top stack item (which will be the public key).
- `OP_HASH160`: Hashes the duplicated public key using SHA256 followed by RIPEMD160, resulting in `PubKeyHash`.
- `<PubKeyHash>`: The 20-byte hash of the recipient's public key.
- `OP_EQUALVERIFY`: Compares the computed `PubKeyHash` with the provided `PubKeyHash`. If they are not equal, the script immediately fails. If they are equal, the `PubKeyHash` is removed from the stack.
- `OP_CHECKSIG`: Verifies the transaction signature against the public key and the transaction data. If the signature is valid, it pushes `TRUE` onto the stack; otherwise, `FALSE`.

#### 2.3.2. Structure of `scriptSig` (Unlocking Script)

The `scriptSig` (also known as the unlocking script or response script) for a P2PKH transaction has the following structure:

`<Signature> <PublicKey>`

- `<Signature>`: The digital signature generated by the sender's private key, proving ownership of the funds.
- `<PublicKey>`: The sender's public key, from which the `PubKeyHash` in the `scriptPubKey` was derived.

#### 2.3.3. Script Evaluation: Unlocking + Locking Concatenation

When a transaction is validated, the `scriptSig` is executed first, followed by the `scriptPubKey`. Conceptually, they are concatenated and executed as a single script. Let's trace the stack flow step-by-step:

**Initial Stack**: Empty

1. **Push `<Signature>`**: The signature from `scriptSig` is pushed onto the stack.
   Stack: `[Signature]`

2. **Push `<PublicKey>`**: The public key from `scriptSig` is pushed onto the stack.
   Stack: `[PublicKey, Signature]`

3. **`OP_DUP`**: Duplicates the top item (`PublicKey`).
   Stack: `[PublicKey, PublicKey, Signature]`

4. **`OP_HASH160`**: Hashes the top `PublicKey` (`SHA256(RIPEMD160(PublicKey))`).
   Stack: `[PubKeyHash, PublicKey, Signature]`

5. **Push `<PubKeyHash>` (from `scriptPubKey`)**: The expected `PubKeyHash` from the locking script is pushed.
   Stack: `[ExpectedPubKeyHash, PubKeyHash, PublicKey, Signature]`

6. **`OP_EQUALVERIFY`**: Compares the top two items (`ExpectedPubKeyHash` and `PubKeyHash`). If they match, both are removed. If not, the script fails.
   Stack: `[PublicKey, Signature]` (assuming match)

7. **`OP_CHECKSIG`**: Takes the `PublicKey` and `Signature` from the stack and verifies the signature against the transaction data. If valid, `TRUE` is pushed.
   Stack: `[TRUE]` (assuming valid signature)

For the transaction to be valid, the final stack must contain a `TRUE` value, and it must be the only item on the stack.

#### 2.3.4. `btcdeb` Validation Steps

`btcdeb` is a powerful tool for debugging Bitcoin scripts. To validate the script execution for a P2PKH transaction, you would typically:

1. **Load the transaction**: `btcdeb -tx <raw_transaction_hex>`
2. **Set the input index**: `input 0`
3. **Execute step-by-step**: Use `step` or `x` to examine the stack and script state after each opcode.

Example `btcdeb` commands for P2PKH:

```bash
# Assuming you have the raw transaction hex for the A->B transaction
btcdeb -tx <raw_tx_hex_A_to_B> -spent_tx <previous_tx_hex_funding_A> -spent_vout <vout_index_funding_A>
input 0
step
# ... continue stepping and observing the stack ...
```

#### 2.3.5. Signature Validation

Signature validation in P2PKH (and generally in Bitcoin) involves `OP_CHECKSIG`. This opcode performs the following checks:

1. **Public Key Recovery**: It uses the provided public key and the signature to verify that the signature was indeed created by the private key corresponding to the public key.
2. **Message Hashing**: It hashes a specific part of the transaction (the 
message that was signed) and compares it against the signature.

If both checks pass, `OP_CHECKSIG` pushes `TRUE` onto the stack, indicating a valid signature and thus valid ownership of the funds.

### 3.3. SegWit Script Analysis (P2SH-P2WPKH)

#### 3.3.1. Structure of `scriptPubKey`

For a P2SH-P2WPKH transaction, the `scriptPubKey` is a P2SH script that locks funds to the hash of a redeem script. Its structure is:

`OP_HASH160 <redeemScriptHash> OP_EQUAL`

- `OP_HASH160`: Hashes the next data push.
- `<redeemScriptHash>`: The 20-byte hash of the `redeemScript`.
- `OP_EQUAL`: Checks if the hash of the data pushed (which should be the `redeemScript`) matches the `redeemScriptHash`. If they match, `TRUE` is pushed onto the stack.

#### 3.3.2. Structure of `redeemScript`

The `redeemScript` for P2SH-P2WPKH is a P2WPKH script. Its structure is:

`0 <PubKeyHash>`

- `0`: This is the witness version byte.
- `<PubKeyHash>`: The 20-byte hash of the public key.

This `redeemScript` is what gets hashed to form the `redeemScriptHash` in the `scriptPubKey`.

#### 3.3.3. Structure of `scriptSig`

For P2SH-P2WPKH, the `scriptSig` contains only the `redeemScript` itself:

`<redeemScript>`

This is pushed onto the stack to satisfy the `OP_HASH160 <redeemScriptHash> OP_EQUAL` in the `scriptPubKey`.

#### 3.3.4. Structure of Witness Data

The actual unlocking data (signature and public key) is moved to the witness field, separate from the `scriptSig`. For P2SH-P2WPKH, the witness stack contains:

`<Signature> <PublicKey>`

- `<Signature>`: The digital signature.
- `<PublicKey>`: The public key.

#### 3.3.5. Explanation of SegWit Characteristics

-   **Why `scriptSig` is smaller**: In P2SH-P2WPKH, the `scriptSig` only contains the `redeemScript` (`0 <PubKeyHash>`), which is significantly smaller than the full `scriptSig` (`<Signature> <PublicKey>`) of a P2PKH transaction. The signature and public key are moved to the witness data.

-   **How witness data is separated**: Witness data is stored in a separate part of the transaction structure, after the traditional transaction fields but before the scriptSig. This separation is key to how SegWit achieves its benefits.

-   **How validation occurs in SegWit**: Validation in SegWit involves two stages:
    1.  **Script Validation (P2SH part)**: The `scriptSig` (containing the `redeemScript`) is executed against the `scriptPubKey`. This verifies that the `redeemScript` matches the `redeemScriptHash` in the `scriptPubKey`.
    2.  **Witness Validation (P2WPKH part)**: If the first stage passes, the `redeemScript` (which is `0 <PubKeyHash>`) is then used to determine how to interpret the witness data. The witness stack (`<Signature> <PublicKey>`) is then executed against a virtual script derived from the `redeemScript` (effectively `OP_DUP OP_HASH160 <PubKeyHash> OP_EQUALVERIFY OP_CHECKSIG`), similar to P2PKH, but using the witness data.

### 4.2. Explanation of SegWit Advantages

-   **Why SegWit transactions are smaller (in virtual size)**: SegWit transactions effectively have a smaller 
virtual size because witness data is discounted. While the raw byte size might not always be significantly smaller, the virtual size (which determines transaction fees) is reduced because witness data is counted as 1/4th its actual size.

-   **Why witness data counts as 1/4 weight**: To incentivize the adoption of SegWit, a new unit called "weight" was introduced. Witness data is given a discount factor of 4, meaning 1 byte of witness data contributes only 1 unit to the transaction's weight, whereas 1 byte of non-witness data contributes 4 units. This effectively makes SegWit transactions cheaper to include in a block.

-   **Fee efficiency**: Due to the 1/4 weight discount for witness data, SegWit transactions consume fewer "block weight units" for the same amount of raw data compared to legacy transactions. This translates directly to lower transaction fees for users, as fees are calculated based on weight units (satoshi/weight unit).

-   **Malleability fix**: One of the primary motivations for SegWit was to fix transaction malleability. In legacy transactions, it was possible for a third party to alter the `scriptSig` (without invalidating the signature) and thus change the transaction ID (TXID) before it was confirmed. This caused issues for applications that relied on unconfirmed TXIDs. By moving the witness data (which includes the signature) outside of the data used to calculate the TXID, SegWit effectively eliminated this form of malleability, making unconfirmed transactions more reliable.
